## 1. 문서 개요

### 1.1 목적

본 문서는 TodoList 제품의 주요 사용자 시나리오를 기술하여, 개발팀·QA팀·디자인팀이 사용자 관점에서 제품 동작을 일관되게 이해하고 검증할 수 있도록 한다.

- 정상 흐름(Happy Path): 대표 페르소나가 일상적으로 수행하는 사용 흐름
- 기능별 상세 시나리오: F-01 ~ F-22 전 기능을 시나리오 형태로 기술
- 예외/오류 시나리오(Edge Case): 실패 케이스 및 경계값 동작 정의
- 화면 전환 흐름: 주요 페이지 간 네비게이션 경로 정의

### 1.2 대상 독자

| 역할 | 활용 목적 |
|---|---|
| 프론트엔드 개발자 | 화면 전환 흐름 및 상태 표현 구현 기준 |
| 백엔드 개발자 | API 응답 및 오류 처리 기준 |

| 기획자 / PO | 기능 범위 및 사용자 경험 검토 |

## 3. 정상 흐름 시나리오 (Happy Path)

### 김철수 페르소나

---

**[SC-A-01] 아침 출근길 — 오늘 할일 확인 및 업무 중 추가**

- 페르소나: 김철수 (27세, 스타트업 마케터)
- 전제 조건: 앱에 계정이 있고 이전에 할일이 등록되어 있다. 스마트폰으로 접속한다.
- 목표: 오늘 처리해야 할 업무 할일을 빠르게 파악하고, 회의 중 발생한 추가 업무를 즉시 등록한다.
- 단계:
  1. 스마트폰 브라우저에서 앱에 접속한다.
  2. 이메일·비밀번호를 입력하고 로그인한다.
  3. 할일 목록 화면(`/`)에서 카테고리 필터를 "업무"로 선택한다.
  4. 상태 필터를 "기간 초과"로 선택하여 마감을 놓친 항목을 먼저 확인한다.
  5. 상태 필터를 "진행 중"으로 바꿔 오늘 처리할 할일 목록을 훑어본다.
  6. 오전 회의 도중 "SNS 광고 소재 검토" 업무가 추가된다.
  7. 상단 할일 추가 버튼을 눌러 제목에 "SNS 광고 소재 검토", 카테고리 "업무", 마감일 오늘 날짜를 입력하고 저장한다.
  8. 목록에 새 할일이 "진행 중" 상태로 추가된 것을 확인한다.
- 기대 결과: 업무 카테고리의 진행 중 및 기간 초과 할일을 파악하고, 신규 할일이 목록에 정상 등록된다.
- 연관 기능: F-02, F-10, F-11

---

**[SC-A-02] 퇴근 후 — 완료 처리 및 불필요 항목 삭제**

- 페르소나: 김철수 (27세, 스타트업 마케터)
- 전제 조건: 로그인 상태. 업무 카테고리에 오늘 처리한 할일 3건이 "진행 중"으로 남아 있다.
- 목표: 오늘 처리한 업무를 완료 상태로 변경하고, 불필요해진 항목을 삭제하여 목록을 정리한다.
- 단계:
  1. 퇴근 후 데스크탑 또는 스마트폰으로 앱에 접속한다.
  2. 할일 목록 화면에서 카테고리 필터를 "업무", 상태 필터를 "진행 중"으로 설정한다.
  3. "SNS 광고 소재 검토" 항목의 완료 버튼을 눌러 완료 처리한다.
  4. 목록에서 해당 항목이 "완료" 상태로 변경된 것을 확인한다.
  5. 더 이상 필요 없는 "임시 메모 — 미팅 전 확인" 항목의 삭제 버튼을 누른다.
  6. 삭제 확인 후 해당 항목이 목록에서 사라진 것을 확인한다.
  7. 남은 2건도 동일하게 완료 처리한다.
  8. 상태 필터를 "완료"로 바꿔 오늘 완료한 전체 업무를 최종 확인한다.
- 기대 결과: 오늘 처리한 업무 3건이 완료 상태로 기록되고, 불필요 항목 1건이 목록에서 영구 삭제된다.
- 연관 기능: F-11, F-13, F-14

### 박소영 페르소나

---

**[SC-B-01] 마트 방문 준비 — 장보기 카테고리 생성 및 항목 추가**

- 페르소나: 박소영 (34세, 전업주부)
- 전제 조건: 로그인 상태. "장보기" 카테고리가 아직 없다.
- 목표: 주말 마트 방문을 위해 장보기 카테고리를 새로 만들고, 구매할 항목들을 할일로 등록한다.
- 단계:
  1. 스마트폰 앱에서 카테고리 관리 화면(`/categories`)으로 이동한다.
  2. "카테고리 추가" 버튼을 눌러 이름에 "장보기"를 입력하고 저장한다.
  3. "장보기" 카테고리가 목록에 추가된 것을 확인한다.
  4. 할일 목록 화면(`/`)으로 이동한다.
  5. 할일 추가 버튼을 눌러 제목 "우유 2L", 카테고리 "장보기"를 입력하고 저장한다.
  6. 같은 방식으로 "계란 30구", "세제", "사과 1봉지" 항목을 차례로 추가한다.
  7. 카테고리 필터를 "장보기"로 설정하여 4개 항목이 모두 "진행 중" 상태로 표시되는지 확인한다.
- 기대 결과: "장보기" 카테고리가 생성되고, 4개의 구매 항목이 해당 카테고리에 등록된다.
- 연관 기능: F-20, F-10, F-11

---

**[SC-B-02] 마트에서 — 완료 처리 및 기간 초과 항목 확인**

- 페르소나: 박소영 (34세, 전업주부)
- 전제 조건: 로그인 상태. "장보기" 카테고리에 4개 항목이 진행 중. 별도로 마감일이 지난 "어린이집 준비물" 항목(가사 카테고리)이 존재한다.
- 목표: 마트에서 장보기 항목을 하나씩 완료 처리하고, 돌아오는 길에 기간 초과 항목을 파악한다.
- 단계:
  1. 스마트폰으로 앱에 접속하여 카테고리 필터를 "장보기"로 설정한다.
  2. "우유 2L" 항목을 장바구니에 담은 후 완료 버튼을 눌러 완료 처리한다.
  3. "계란 30구", "세제", "사과 1봉지"도 순서대로 완료 처리한다.
  4. 장보기 카테고리 4개 항목이 모두 "완료" 상태가 된 것을 확인한다.
  5. 카테고리 필터를 전체 해제하고, 상태 필터를 "기간 초과"로 변경한다.
  6. "어린이집 준비물" 항목이 기간 초과 상태로 표시되는 것을 확인한다.
  7. 해당 항목의 마감일을 내일로 수정하여 "진행 중" 상태로 전환한다.
- 기대 결과: 장보기 4개 항목이 완료 처리되고, 기간 초과 항목이 파악되어 마감일이 수정된다.
- 연관 기능: F-11, F-13, F-12

## 4. 기능별 상세 시나리오

### 4.1 인증 기능

---

**[SC-A-10] 신규 회원 가입**

- 페르소나: 김철수
- 전제 조건: 앱에 가입된 계정이 없다. `/register` 화면이 열려 있다.
- 목표: 이메일과 비밀번호로 새 계정을 생성한다.
- 단계:
  1. `/register` 화면에서 이메일 입력란에 미사용 이메일을 입력한다.
  2. 비밀번호 입력란에 8자 이상의 비밀번호를 입력한다.
  3. 비밀번호 확인란에 동일한 비밀번호를 재입력한다.
  4. "가입하기" 버튼을 클릭한다.
  5. 가입 성공 메시지가 표시되고 `/login` 화면으로 이동한다.
- 기대 결과: 계정이 생성되고(201), 로그인 화면으로 리다이렉트된다.
- 연관 기능: F-01

---

**[SC-A-11] 이메일·비밀번호로 로그인**

- 페르소나: 김철수
- 전제 조건: 이미 가입된 계정이 존재한다. `/login` 화면이 열려 있다.
- 목표: 이메일·비밀번호로 인증하여 서비스에 접근한다.
- 단계:
  1. `/login` 화면에서 가입된 이메일을 입력한다.
  2. 올바른 비밀번호를 입력한다.
  3. "로그인" 버튼을 클릭한다.
  4. Access Token과 Refresh Token이 클라이언트에 저장된다.
  5. 할일 목록 화면(`/`)으로 자동 이동한다.
- 기대 결과: JWT 토큰이 발급되고(200), 할일 목록 화면으로 이동한다.
- 연관 기능: F-02

---

**[SC-A-12] 로그아웃**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. 할일 목록 화면(`/`)에 있다.
- 목표: 세션을 종료하고 로그인 화면으로 이동한다.
- 단계:
  1. 상단 내비게이션에서 "로그아웃" 버튼을 클릭한다.
  2. 클라이언트에 저장된 Access Token과 Refresh Token이 삭제된다.
  3. `/login` 화면으로 리다이렉트된다.
  4. 브라우저 뒤로가기로 이전 화면 접근 시 다시 `/login`으로 리다이렉트된다.
- 기대 결과: 토큰이 클라이언트에서 제거되고, 이후 인증이 필요한 모든 요청은 차단된다.
- 연관 기능: F-03

---

**[SC-A-13] Access Token 자동 갱신**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. Access Token이 만료(1시간 경과)되었으나 Refresh Token(7일)은 유효하다.
- 목표: 사용자가 인식하지 못하는 상태에서 세션이 자동으로 연장된다.
- 단계:
  1. 김철수가 할일 목록 조회 API를 요청한다.
  2. 서버가 401 응답을 반환한다(Access Token 만료).
  3. 클라이언트가 Refresh Token으로 새 Access Token 발급 요청을 자동으로 전송한다.
  4. 서버가 새 Access Token을 발급한다.
  5. 클라이언트가 새 Access Token으로 원래 요청을 재시도한다.
  6. 할일 목록이 정상적으로 로드된다.
- 기대 결과: 사용자 개입 없이 토큰이 갱신되고, 화면이 정상 표시된다.
- 연관 기능: F-04

### 4.2 할일 기능

---

**[SC-A-20] 제목만 입력하여 할일 빠르게 생성**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. 할일 목록 화면(`/`)에 있다.
- 목표: 최소 정보(제목만)로 할일을 빠르게 등록한다.
- 단계:
  1. 할일 추가 버튼을 클릭한다.
  2. 제목 입력란에 "팀 주간 보고서 작성" (최대 100자 이내)을 입력한다.
  3. 설명·카테고리·마감일은 입력하지 않는다.
  4. "저장" 버튼을 클릭한다.
  5. 새 할일이 목록에 "진행 중" 상태로 추가된다.
- 기대 결과: 제목만 가진 할일이 생성되고(201), 마감일이 null이므로 "진행 중" 상태로 표시된다.
- 연관 기능: F-10

---

**[SC-A-21] 카테고리·마감일 포함 할일 생성**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. "업무" 카테고리가 존재한다.
- 목표: 마감일과 카테고리를 지정하여 일정 관리가 가능한 할일을 등록한다.
- 단계:
  1. 할일 추가 버튼을 클릭한다.
  2. 제목에 "경쟁사 광고 분석 리포트"를 입력한다.
  3. 설명에 "4월 캠페인 기간 데이터 포함"을 입력한다.
  4. 카테고리 드롭다운에서 "업무"를 선택한다.
  5. 마감일에 3일 후 날짜를 선택한다.
  6. "저장" 버튼을 클릭한다.
  7. 목록에 "진행 중" 상태로 해당 할일이 표시된다.
- 기대 결과: 제목·설명·카테고리·마감일이 모두 저장되고(201), 마감일 > 현재이므로 "진행 중"으로 표시된다.
- 연관 기능: F-10

---

**[SC-A-22] 상태 필터로 기간 초과 할일 조회**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. 마감일이 어제인 미완료 할일 2건이 존재한다.
- 목표: 기간이 지난 할일을 빠르게 식별하여 우선 처리한다.
- 단계:
  1. 할일 목록 화면에서 상태 필터 드롭다운을 "기간 초과"로 선택한다.
  2. 마감일이 지난 미완료 할일 2건만 목록에 표시된다.
  3. 각 항목의 마감일 표시가 강조(예: 붉은색)되어 있음을 확인한다.
- 기대 결과: 상태 = 기간 초과인 항목만 필터링되어 표시된다. 본인 데이터만 포함된다.
- 연관 기능: F-11

---

**[SC-A-23] 카테고리 필터로 업무 할일만 조회**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. "업무" 카테고리와 "개인" 카테고리 할일이 혼재한다.
- 목표: 업무 관련 할일만 모아서 확인한다.
- 단계:
  1. 할일 목록 화면에서 카테고리 필터 드롭다운을 "업무"로 선택한다.
  2. "업무" 카테고리에 속한 할일만 목록에 표시된다.
  3. "개인" 카테고리 할일과 카테고리 미지정 할일은 목록에서 제외된다.
- 기대 결과: 선택한 카테고리에 속한 할일만 표시된다. 타인의 데이터는 포함되지 않는다.
- 연관 기능: F-11

---

**[SC-A-24] 할일 제목·마감일 수정**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. "팀 주간 보고서 작성" 할일이 존재한다.
- 목표: 할일 제목을 수정하고 마감일을 추가한다.
- 단계:
  1. 해당 할일 카드의 수정 버튼을 클릭한다.
  2. 제목을 "팀 주간 보고서 작성 (초안)"으로 변경한다.
  3. 마감일에 오늘 저녁 날짜를 선택한다.
  4. "저장" 버튼을 클릭한다.
  5. 목록에서 수정된 제목과 마감일이 반영된 것을 확인한다.
- 기대 결과: 제목과 마감일이 업데이트되고, 수정일이 현재 시각으로 갱신된다.
- 연관 기능: F-12

---

**[SC-A-25] 할일 완료 처리**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. "경쟁사 광고 분석 리포트" 할일이 "진행 중" 상태이다.
- 목표: 완료한 업무를 완료 상태로 표시한다.
- 단계:
  1. 해당 할일 카드의 완료 버튼(체크박스 또는 버튼)을 클릭한다.
  2. 완료 여부가 true로 변경된다.
  3. 해당 할일의 상태가 "완료"로 전환되고, 시각적으로 구분(예: 취소선, 흐리게)된다.
- 기대 결과: 완료 여부 = true로 저장되고, 상태가 즉시 "완료"로 전환된다.
- 연관 기능: F-13

---

**[SC-A-26] 완료된 할일 완료 취소 (진행 중으로 복귀)**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. "경쟁사 광고 분석 리포트" 할일이 "완료" 상태이고 마감일이 아직 남아 있다.
- 목표: 실수로 완료 처리한 할일을 진행 중 상태로 되돌린다.
- 단계:
  1. 상태 필터를 "완료"로 설정하여 해당 할일을 찾는다.
  2. 완료 버튼을 다시 클릭하여 완료 취소한다.
  3. 완료 여부가 false로 변경된다.
  4. 마감일이 현재 시각보다 이후이므로 상태가 "진행 중"으로 재분류된다.
- 기대 결과: 완료 여부 = false로 저장되고, 마감일 조건에 따라 "진행 중"으로 표시된다.
- 연관 기능: F-13

---

**[SC-A-27] 할일 삭제**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. "임시 메모 — 미팅 전 확인" 할일이 존재한다.
- 목표: 더 이상 필요 없는 할일을 영구 삭제한다.
- 단계:
  1. 해당 할일 카드의 삭제 버튼을 클릭한다.
  2. 삭제 확인 다이얼로그가 표시된다.
  3. "삭제" 버튼을 클릭하여 확인한다.
  4. 해당 할일이 목록에서 즉시 사라진다.
- 기대 결과: 할일이 물리 삭제(Hard Delete)되고, 목록에서 제거된다. 복구 불가.
- 연관 기능: F-14

### 4.3 카테고리 기능

---

**[SC-B-10] 카테고리 생성**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. 카테고리 관리 화면(`/categories`)에 있다. "육아" 카테고리가 아직 없다.
- 목표: 새로운 "육아" 카테고리를 생성한다.
- 단계:
  1. 카테고리 관리 화면에서 "카테고리 추가" 버튼을 클릭한다.
  2. 이름 입력란에 "육아"를 입력한다. (50자 이내)
  3. "저장" 버튼을 클릭한다.
  4. 카테고리 목록에 "육아"가 추가된 것을 확인한다.
- 기대 결과: 카테고리가 생성되고(201), 현재 사용자에게만 귀속된다.
- 연관 기능: F-20

---

**[SC-B-11] 카테고리 이름 수정**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. "가사" 카테고리가 존재한다.
- 목표: "가사" 카테고리 이름을 "집안일"로 변경한다.
- 단계:
  1. 카테고리 관리 화면에서 "가사" 카테고리 옆 수정 버튼을 클릭한다.
  2. 이름을 "집안일"로 변경한다.
  3. "저장" 버튼을 클릭한다.
  4. 카테고리 목록에서 "가사"가 "집안일"로 변경된 것을 확인한다.
  5. 할일 목록 화면으로 이동하여, 기존 "가사" 카테고리 할일들의 카테고리명이 "집안일"로 표시되는지 확인한다.
- 기대 결과: 카테고리 이름이 변경되고, 해당 카테고리에 속한 모든 할일에 변경된 이름이 반영된다.
- 연관 기능: F-21

---

**[SC-B-12] 카테고리 삭제 — 소속 할일 보존 확인**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. "장보기" 카테고리와 해당 카테고리 소속 할일 3건이 존재한다.
- 목표: "장보기" 카테고리를 삭제하되, 소속 할일 3건은 유지되는지 확인한다.
- 단계:
  1. 카테고리 관리 화면에서 "장보기" 카테고리 옆 삭제 버튼을 클릭한다.
  2. 삭제 확인 안내("소속 할일의 카테고리 분류가 해제됩니다" 등)가 표시된다.
  3. "삭제" 버튼을 클릭한다.
  4. "장보기" 카테고리가 목록에서 사라진다.
  5. 할일 목록 화면으로 이동하여, 이전 "장보기" 소속 할일 3건이 카테고리 미지정 상태로 존재하는지 확인한다.
- 기대 결과: 카테고리가 삭제되고, 소속 할일의 카테고리 ID는 null 처리되나 할일 자체는 보존된다(DR-06).
- 연관 기능: F-22

## 5. 예외/오류 시나리오 (Edge Case)

### 5.1 인증 오류

---

**[SC-E-01] 중복 이메일로 회원 가입 시도**

- 페르소나: 해당 없음 (신규 가입 시도 사용자)
- 전제 조건: "test@example.com" 이메일로 이미 가입된 계정이 존재한다.
- 목표: (사용자 실수) 이미 사용 중인 이메일로 재가입을 시도한다.
- 단계:
  1. `/register` 화면에서 "test@example.com"을 이메일 입력란에 입력한다.
  2. 비밀번호를 8자 이상으로 입력한다.
  3. "가입하기" 버튼을 클릭한다.
  4. 서버가 409 Conflict를 반환한다.
  5. 화면에 "이미 사용 중인 이메일입니다" 오류 메시지가 표시된다.
- 기대 결과: 계정이 생성되지 않고, 이메일 중복 오류 메시지가 표시된다.
- 연관 기능: F-01

---

**[SC-E-02] 비밀번호 8자 미만으로 회원 가입 시도**

- 페르소나: 해당 없음
- 전제 조건: `/register` 화면이 열려 있다.
- 목표: (사용자 실수) 짧은 비밀번호로 가입을 시도한다.
- 단계:
  1. 미사용 이메일을 입력한다.
  2. 비밀번호 입력란에 "pass123" (7자)을 입력한다.
  3. "가입하기" 버튼을 클릭한다.
  4. 서버가 400 Bad Request를 반환한다.
  5. 화면에 "비밀번호는 8자 이상이어야 합니다" 오류 메시지가 표시된다.
- 기대 결과: 계정이 생성되지 않고, 비밀번호 길이 오류 메시지가 표시된다.
- 연관 기능: F-01

---

**[SC-E-03] 잘못된 비밀번호로 로그인 시도**

- 페르소나: 김철수
- 전제 조건: "test@example.com" 계정이 존재한다. `/login` 화면이 열려 있다.
- 목표: (사용자 실수) 틀린 비밀번호로 로그인을 시도한다.
- 단계:
  1. 이메일에 "test@example.com"을 입력한다.
  2. 비밀번호에 틀린 값을 입력한다.
  3. "로그인" 버튼을 클릭한다.
  4. 서버가 401 Unauthorized를 반환한다.
  5. 화면에 "이메일 또는 비밀번호가 올바르지 않습니다" 오류 메시지가 표시된다.
- 기대 결과: 로그인이 거부되고, 401 오류 메시지가 표시된다. 토큰이 발급되지 않는다.
- 연관 기능: F-02

---

**[SC-E-04] 가입되지 않은 이메일로 로그인 시도**

- 페르소나: 해당 없음
- 전제 조건: `/login` 화면이 열려 있다.
- 목표: (사용자 실수) 존재하지 않는 계정으로 로그인을 시도한다.
- 단계:
  1. 가입되지 않은 이메일 "unknown@example.com"을 입력한다.
  2. 임의 비밀번호를 입력한다.
  3. "로그인" 버튼을 클릭한다.
  4. 서버가 401 Unauthorized를 반환한다.
  5. 화면에 오류 메시지가 표시된다.
- 기대 결과: 로그인이 거부된다. 보안을 위해 "이메일이 존재하지 않습니다" 등 구체적 이유를 노출하지 않는다.
- 연관 기능: F-02

---

**[SC-E-05] 미인증 상태에서 보호 페이지 접근**

- 페르소나: 해당 없음 (미로그인 사용자)
- 전제 조건: 브라우저에 Access Token이 없다(로그아웃 또는 미로그인 상태).
- 목표: (경계값) 인증 없이 보호된 페이지에 직접 URL 접근을 시도한다.
- 단계:
  1. 브라우저 주소창에 직접 `/` (할일 목록)을 입력한다.
  2. 클라이언트가 인증 토큰 없음을 감지한다.
  3. `/login` 화면으로 자동 리다이렉트된다.
  4. 동일하게 `/categories` URL을 직접 입력하면 다시 `/login`으로 리다이렉트된다.
- 기대 결과: 미인증 사용자는 인증 필요 화면에 접근할 수 없고, 항상 `/login`으로 이동된다.
- 연관 기능: F-02, F-04

### 5.2 할일 유효성 오류

---

**[SC-E-06] 제목 없이 할일 생성 시도**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. 할일 추가 폼이 열려 있다.
- 목표: (경계값) 필수 필드인 제목을 비운 채 저장을 시도한다.
- 단계:
  1. 할일 추가 폼에서 제목 입력란을 비워둔다.
  2. 설명에 "내용만 입력함"을 입력한다.
  3. "저장" 버튼을 클릭한다.
  4. 서버가 400 Bad Request를 반환한다.
  5. "제목은 필수 항목입니다" 오류 메시지가 표시된다.
- 기대 결과: 할일이 생성되지 않고, 제목 필수 오류가 표시된다.
- 연관 기능: F-10

---

**[SC-E-07] 제목 101자 초과 할일 생성 시도**

- 페르소나: 김철수
- 전제 조건: 로그인 상태. 할일 추가 폼이 열려 있다.
- 목표: (경계값) 최대 길이(100자)를 초과하는 제목으로 저장을 시도한다.
- 단계:
  1. 할일 추가 폼의 제목 입력란에 101자 문자열을 입력한다.
  2. "저장" 버튼을 클릭한다.
  3. 서버가 400 Bad Request를 반환한다.
  4. "제목은 100자 이내로 입력해야 합니다" 오류 메시지가 표시된다.
- 기대 결과: 할일이 생성되지 않고, 제목 길이 초과 오류가 표시된다. (프론트엔드에서 사전 차단 권장)
- 연관 기능: F-10

---

### 5.3 권한 오류

---

**[SC-E-08] 타인의 할일 수정 시도**

- 페르소나: 해당 없음 (악의적 사용자 또는 오작동)
- 전제 조건: 사용자 A, 사용자 B가 각각 로그인된 별도 세션. 사용자 A가 사용자 B의 할일 ID를 알고 있다.
- 목표: (보안 경계값) 사용자 A가 사용자 B의 할일 ID로 수정 API를 직접 호출한다.
- 단계:
  1. 사용자 A의 Access Token으로 사용자 B 소유 할일 ID에 수정 요청(PATCH/PUT)을 전송한다.
  2. 서버가 소유권 검증을 수행한다(DR-02).
  3. 서버가 403 Forbidden과 오류 메시지를 반환한다.
- 기대 결과: 수정이 거부되고 403이 반환된다. 사용자 B의 데이터는 변경되지 않는다.
- 연관 기능: F-12

---

**[SC-E-09] 타인의 할일 삭제 시도**

- 페르소나: 해당 없음 (악의적 사용자 또는 오작동)
- 전제 조건: 사용자 A가 사용자 B의 할일 ID를 알고 있다.
- 목표: (보안 경계값) 사용자 A가 사용자 B의 할일 ID로 삭제 API를 직접 호출한다.
- 단계:
  1. 사용자 A의 Access Token으로 사용자 B 소유 할일 ID에 삭제 요청(DELETE)을 전송한다.
  2. 서버가 소유권 검증을 수행한다(DR-02).
  3. 서버가 403 Forbidden과 오류 메시지를 반환한다.
- 기대 결과: 삭제가 거부되고 403이 반환된다. 사용자 B의 데이터는 삭제되지 않는다.
- 연관 기능: F-14

---

### 5.4 상태 전환 엣지 케이스

---

**[SC-E-10] 완료 취소 시 마감일이 이미 경과된 경우**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. "어린이집 준비물 챙기기" 할일이 "완료" 상태이고, 마감일이 어제(현재 시각보다 이전)이다.
- 목표: (상태 전환 엣지 케이스) 완료를 취소했을 때 마감일이 지났으므로 "진행 중"이 아닌 "기간 초과"로 분류되어야 한다.
- 단계:
  1. 상태 필터를 "완료"로 설정하여 해당 할일을 찾는다.
  2. 완료 버튼을 클릭하여 완료 취소한다.
  3. 완료 여부가 false로 변경된다.
  4. 마감일(어제) ≤ 현재 시각이므로 상태가 "기간 초과"로 재분류된다.
  5. 상태 필터를 "기간 초과"로 변경하면 해당 할일이 표시된다.
- 기대 결과: 완료 취소 후 마감일 조건에 따라 "기간 초과"로 표시된다(DR-08).
- 연관 기능: F-13

---

**[SC-E-14] 마감일이 null인 할일의 상태 확인**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. "언젠가 읽을 책 목록" 할일이 마감일 없이(null) 생성되어 있다.
- 목표: (경계값) 마감일이 null인 미완료 할일이 항상 "진행 중"으로 분류되는지 확인한다.
- 단계:
  1. 할일 목록 화면에서 상태 필터를 "기간 초과"로 설정한다.
  2. "언젠가 읽을 책 목록" 할일이 목록에 표시되지 않는다.
  3. 상태 필터를 "진행 중"으로 변경한다.
  4. "언젠가 읽을 책 목록" 할일이 "진행 중" 상태로 표시된다.
- 기대 결과: 마감일 = null인 미완료 할일은 시간이 아무리 지나도 "기간 초과"가 되지 않고 "진행 중"으로 유지된다(DR-07).
- 연관 기능: F-11

---

### 5.5 카테고리 오류

---

**[SC-E-11] 중복 이름으로 카테고리 생성 시도**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. "육아" 카테고리가 이미 존재한다.
- 목표: (경계값) 동일한 이름으로 카테고리를 다시 생성하려 시도한다.
- 단계:
  1. 카테고리 관리 화면에서 "카테고리 추가" 버튼을 클릭한다.
  2. 이름에 "육아"를 입력한다.
  3. "저장" 버튼을 클릭한다.
  4. 서버가 409 Conflict를 반환한다.
  5. "이미 사용 중인 카테고리 이름입니다" 오류 메시지가 표시된다.
- 기대 결과: 카테고리가 중복 생성되지 않고 409 오류 메시지가 표시된다(DR-05).
- 연관 기능: F-20

---

**[SC-E-12] 중복 이름으로 카테고리 수정 시도**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. "육아"와 "집안일" 카테고리가 모두 존재한다.
- 목표: (경계값) "집안일" 카테고리 이름을 이미 존재하는 "육아"로 변경하려 시도한다.
- 단계:
  1. 카테고리 관리 화면에서 "집안일" 카테고리의 수정 버튼을 클릭한다.
  2. 이름을 "육아"로 변경한다.
  3. "저장" 버튼을 클릭한다.
  4. 서버가 409 Conflict를 반환한다.
  5. "이미 사용 중인 카테고리 이름입니다" 오류 메시지가 표시된다.
- 기대 결과: 카테고리 이름이 변경되지 않고 409 오류 메시지가 표시된다.
- 연관 기능: F-21

---

**[SC-E-13] 카테고리 이름 50자 초과 시도**

- 페르소나: 박소영
- 전제 조건: 로그인 상태. 카테고리 추가 폼이 열려 있다.
- 목표: (경계값) 최대 길이(50자)를 초과하는 이름으로 카테고리 생성을 시도한다.
- 단계:
  1. 카테고리 이름 입력란에 51자 문자열을 입력한다.
  2. "저장" 버튼을 클릭한다.
  3. 서버가 400 Bad Request를 반환한다.
  4. "카테고리 이름은 50자 이내로 입력해야 합니다" 오류 메시지가 표시된다.
- 기대 결과: 카테고리가 생성되지 않고, 이름 길이 초과 오류가 표시된다. (프론트엔드에서 사전 차단 권장)
- 연관 기능: F-20

## 6. 화면 전환 흐름

### 6.1 미인증 사용자 접근 흐름

```
[브라우저 URL 직접 입력]
        │
        ▼
  인증 토큰 확인
        │
   ┌────┴─────┐
없음             있음 (유효)
   │               │
   ▼               ▼
/login          요청한 화면으로 진행
(리다이렉트)
```

---

### 6.2 로그인 흐름

```
/login
   │
   ├─ 로그인 성공 ──────────────────────────────▶ / (할일 목록)
   │
   ├─ 로그인 실패 (401) ────────────────────────▶ /login (오류 메시지 표시)
   │
   └─ "회원 가입" 링크 클릭 ───────────────────▶ /register
```

---

### 6.3 회원 가입 흐름

```
/register
   │
   ├─ 가입 성공 (201) ──────────────────────────▶ /login (성공 안내 후 이동)
   │
   ├─ 이메일 중복 (409) ────────────────────────▶ /register (오류 메시지 표시)
   │
   ├─ 비밀번호 미충족 (400) ────────────────────▶ /register (오류 메시지 표시)
   │
   └─ "로그인으로 돌아가기" 링크 클릭 ─────────▶ /login
```

---

### 6.4 주요 화면 내 네비게이션

```
/ (할일 목록)
   │
   ├─ "카테고리 관리" 메뉴 클릭 ───────────────▶ /categories
   │
   ├─ 할일 추가 버튼 클릭 ──────────────────────▶ 할일 추가 폼 (모달 또는 인라인)
   │
   └─ 로그아웃 버튼 클릭 ───────────────────────▶ /login

/categories
   │
   ├─ "할일 목록" 메뉴 클릭 ────────────────────▶ /
   │
   └─ 로그아웃 버튼 클릭 ───────────────────────▶ /login
```

---

### 6.5 카테고리 삭제 후 할일 목록 흐름

```
/categories
   │
   └─ "장보기" 카테고리 삭제 확인
           │
           ▼
     카테고리 삭제 완료
     - "장보기" 카테고리: 목록에서 제거
     - 소속 할일 3건: 카테고리 ID → null (보존)
           │
           ▼
   / (할일 목록) 이동 시
     - 이전 "장보기" 소속 할일 3건:
       카테고리 = "미지정"으로 표시
     - 카테고리 필터 드롭다운에서
       "장보기" 항목 사라짐
```

### 6.6 Access Token 만료 및 자동 갱신 흐름

```
[API 요청]
    │
    ▼
서버 응답 401
(Access Token 만료)
    │
    ▼
Refresh Token 유효성 확인
    │
   ┌┴─────────────────┐
유효                   만료(7일 경과)
   │                       │
   ▼                       ▼

   │                   (재로그인 필요 안내)

   │
   ▼
정상 응답 반환 → 화면 표시
```

### 6.7 완료 취소 후 상태 재분류 흐름
