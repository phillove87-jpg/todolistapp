## 1. 제품 개요

### 1.1 제품 비전

카테고리 분류와 마감일 관리를 직관적으로 제공하는 개인 할일 관리 웹 애플리케이션.  
사용자는 회원 가입 후 언제 어디서나 반응형 웹 브라우저를 통해 자신의 할일을 체계적으로 등록·관리·추적할 수 있다.

### 1.2 핵심 차별점

| 기존 앱의 문제 | TodoList의 해결 방식 |
|---|---|
| 카테고리 분류가 없거나 고정되어 있음 | 사용자가 카테고리를 자유롭게 생성·수정·삭제 가능 |
| 마감일 표현이 없거나 단순 날짜만 표시 | 마감일 기반 상태(진행 중/기간 초과/완료) 자동 계산 및 시각적 구분 |

| 주요 사용 시나리오 | 아침에 오늘 할일 확인 → 업무 중 새 할일 추가 → 퇴근 후 완료 처리 |

## 4. 범위 정의 (Scope)

### 4.1 Phase 1 (2026-04-28 ~ 2026-04-30) — 현재 버전

| 할일 관리 | 할일 생성·조회·수정(완료 처리 포함)·삭제 | UC-03, UC-04 |
| 카테고리 관리 | 카테고리 생성·수정·삭제, 카테고리별 필터링 | UC-05 |

### 4.2 Phase 2 (일정 미정) — 향후 버전

| 기능 | 설명 |

## 5. 기능 요구사항

> 상세 도메인 규칙 및 인수 조건은 [도메인 정의서 Section 5](./1-domain-definition.md) 참조

### 5.1 사용자 정보 관리

| ID | 기능 | 요구사항 |
|---|---|---|
| F-01 | 회원 가입 | 이메일(고유값) + 비밀번호(8자 이상)로 계정 생성. 중복 이메일 가입 불가 |
| F-02 | 로그인 | 이메일 + 비밀번호 검증 후 JWT 토큰 발급. 실패 시 401 반환 |
| F-03 | 로그아웃 | 클라이언트 측 토큰 제거. 이후 인증 필요 요청은 401 반환 |
| F-04 | 인증 유지 | Access Token 만료 시 Refresh Token으로 자동 갱신 |

### 5.2 할일 관리

| ID | 기능 | 요구사항 |
|---|---|---|
| F-10 | 할일 생성 | 제목(필수, 100자 이내), 설명(선택), 카테고리(선택), 마감일(선택) 입력 |
| F-11 | 할일 목록 조회 | 상태(진행 중/완료/기간 초과) 및 카테고리로 필터링. 본인 데이터만 조회 |
| F-12 | 할일 수정 | 제목·설명·마감일·카테고리·완료 여부 변경 가능 |
| F-13 | 완료 처리 / 완료 취소 | 완료 여부 토글. 취소 시 마감일 기준으로 진행 중 또는 기간 초과 재분류 |
| F-14 | 할일 삭제 | 즉시 물리 삭제(Hard Delete). 타인 데이터 삭제 시도 시 403 반환 |

**할일 상태 계산 규칙 (도메인 정의서 DR-03, DR-04, DR-07 기반)**

```
완료 여부 = true              → 완료
완료 여부 = false, 마감일 = null  → 진행 중
완료 여부 = false, 마감일 > 현재  → 진행 중
완료 여부 = false, 마감일 ≤ 현재  → 기간 초과
```

### 5.3 카테고리 관리

| ID | 기능 | 요구사항 |
|---|---|---|
| F-20 | 카테고리 생성 | 이름(필수, 50자 이내). 동일 사용자 내 이름 중복 불가 |
| F-21 | 카테고리 수정 | 이름 변경 가능. 중복 이름으로 변경 시 409 반환 |
| F-22 | 카테고리 삭제 | 삭제 시 소속 할일의 카테고리 ID는 null 처리. 할일 자체는 보존 |

## 6. 비기능적 요구사항 (NFR)

| 항목 | 요구사항 |
|---|---|
| **보안** | 비밀번호 bcrypt 해시 저장. JWT HS-512 서명. 모든 API 인증 강제 (DR-01) |
| **응답 속도** | 일반 CRUD API 응답 500ms 이하 (로컬 환경 기준) |
| **반응형** | 뷰포트 360px(모바일) ~ 1440px(데스크탑) 정상 렌더링 |
| **브라우저 지원** | Chrome, Edge, Safari, Firefox 최신 버전 |
| **데이터 격리** | 사용자는 자신의 데이터에만 접근 가능. 서버에서 소유권 강제 검증 (DR-02) |
| **에러 처리** | 모든 API 오류는 HTTP 상태코드 + 한국어 메시지 JSON으로 반환 |

## 7. 기술 스택 및 아키텍처

### 7.1 아키텍처 개요

```
[Browser] ──HTTP/REST──▶ [Backend API] ──SQL──▶ [PostgreSQL]
   React 19                 Express 5              DB
   (Responsive Web)         Node 24
```

### 7.2 기술 스택 상세

| 레이어 | 기술 | 버전 | 비고 |
|---|---|---|---|
| **Frontend** | React | 19 | JavaScript (TypeScript 미사용) |
| | TanStack Query | 최신 | 서버 상태 관리 (API 캐싱·동기화) |
| | Zustand | 최신 | 전역 클라이언트 상태 관리 |
| | Tailwind CSS | 최신 | 유틸리티 기반 스타일링 |
| **Backend** | Node.js | 24 | JavaScript (TypeScript 미사용) |
| | Express | 5 | REST API 프레임워크 |
| | pg | 최신 | PostgreSQL 클라이언트 라이브러리 |
| **DB** | PostgreSQL | 최신 LTS | 관계형 데이터베이스 |
| **인증** | JWT | — | HS-512 알고리즘, Access + Refresh Token |

### 7.3 인증 흐름

```
1. 로그인 요청 → 서버에서 이메일/비밀번호 검증
2. 검증 성공 → HS-512 서명된 Access Token(1h) + Refresh Token(7d) 발급
3. 이후 요청 → Authorization: Bearer <accessToken> 헤더 포함
4. Access Token 만료 → Refresh Token으로 재발급 요청
5. Refresh Token 만료 → 재로그인 필요
```
