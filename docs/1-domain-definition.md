## 1. 문제 정의

개인 할일 관리 앱 중 카테고리 분류와 종료 시간(마감일)을 함께 표현할 수 있는 도구가 없어, 일정을 체계적으로 관리하기 어렵다.

---

## 2. 핵심 도메인 개념

### 2.1 엔티티 관계 요약

```
User 1 ──< Todo >── (0..1) Category
User 1 ──< Category
```

- 한 사용자(User)는 여러 할일(Todo)을 가진다. (1:N)
- 한 사용자(User)는 여러 카테고리(Category)를 가진다. (1:N)
- 한 할일(Todo)은 하나의 카테고리에 속하거나, 미지정(null) 상태일 수 있다. (N:0..1)
- 카테고리는 해당 사용자에게만 속하며 타 사용자와 공유되지 않는다.

### 2.2 엔티티 및 속성 상세

| 엔티티 | 속성 | 타입 | 제약 조건 | 설명 |
|---|---|---|---|---|
| **User** | ID | UUID | PK, Not Null | 사용자 식별자 |
| | 이메일 | String | Unique, Email, Not Null | 로그인 계정 |
| | 비밀번호 | String | Min 8 chars, Not Null | 암호화 저장 (평문 저장 금지) |
| | 가입일 | DateTime | Not Null | 계정 생성 시점 |
| **Todo** | ID | UUID | PK, Not Null | 할일 식별자 |
| | 제목 | String | Max 100, Not Null | 할일 제목 |
| | 설명 | Text | Nullable | 상세 내용 (선택 사항) |
| | 카테고리 ID | UUID | FK → Category.ID, Nullable | 소속 카테고리 (삭제 시 null) |
| | 마감일 | DateTime | Nullable | 완료 목표 시점 (미입력 허용) |
| | 완료 여부 | Boolean | Default: false, Not Null | 완료 상태 플래그 |
| | 생성일 | DateTime | Not Null | 데이터 생성 시점 |
| | 수정일 | DateTime | Not Null | 최종 수정 시점 |
| **Category** | ID | UUID | PK, Not Null | 카테고리 식별자 |
| | 이름 | String | Max 50, Not Null | 카테고리 명칭 |
| | 사용자 ID | UUID | FK → User.ID, Not Null | 소유 사용자 식별자 |

### 2.3 도메인 규칙 (Domain Rules)

| # | 규칙 설명 | 적용 기능 | 비고 |
|---|---|---|---|
| DR-01 | 모든 기능은 인증된 사용자만 사용할 수 있다. | UC-01, UC-02, UC-03, UC-04, UC-05 전체 | JWT Bearer 토큰 검증 |
| DR-02 | 사용자는 자신의 데이터(할일, 카테고리)만 조회·수정·삭제할 수 있다. | UC-03, UC-04, UC-05 | 소유권 검증은 서버에서 강제 |
| DR-03 | 할일은 **진행 중 / 완료 / 기간 초과** 세 가지 상태로 구분된다. | UC-04 | Section 3 참조 |
| DR-04 | 마감일이 현재 시각을 지난 미완료 할일은 자동으로 **기간 초과** 상태가 된다. | UC-04 | 상태는 DB 컬럼이 아닌 시스템 계산값 |
| DR-05 | 동일 사용자 내에서 카테고리 이름은 중복될 수 없다. | UC-05 | 다른 사용자 간 동일 이름은 허용 |
| DR-06 | 카테고리 삭제 시 해당 카테고리 소속 할일의 카테고리 ID는 null로 설정된다. | UC-05 | 할일 자체는 삭제되지 않음 (ADR-01 참조) |
| DR-07 | 마감일이 null인 할일은 완료 여부와 무관하게 **진행 중** 상태로 취급된다. | UC-03, UC-04 | null 마감일 = 기간 제한 없음 |
| DR-08 | 완료 상태의 할일은 완료 여부를 false로 변경하여 **진행 중** 상태로 되돌릴 수 있다. | UC-03 | 단, 마감일이 이미 지난 경우 기간 초과로 재분류됨 |

## 3. 할일 상태 정의 (State Logic)

```
Todo.상태 결정 로직

IF 완료 여부 = true
  → 완료

ELSE IF 마감일 = null
  → 진행 중  (DR-07: 마감일 없음 = 기간 제한 없음)

ELSE IF 마감일 > 현재 시각
  → 진행 중

ELSE  (마감일 ≤ 현재 시각)
  → 기간 초과
```

> 상태(status)는 DB에 저장되지 않으며, 조회 시점에 위 로직으로 계산된다.

## 4. 인증 메커니즘

| 항목 | 내용 |
|---|---|
| 방식 | JWT (JSON Web Token) Bearer 토큰 |
| 발급 시점 | 로그인 성공 시 |
| 만료 시간 | Access Token: 1시간 / Refresh Token: 7일 |
| 갱신 방법 | Refresh Token으로 새 Access Token 발급 |
| 전달 방식 | HTTP Authorization 헤더 (`Bearer <token>`) |
| 비밀번호 저장 | bcrypt 해시 후 저장 (평문 저장 금지) |

## 5. 주요 기능 및 인수 조건 (Use Cases & AC)

### UC-01 회원 가입

**연관 규칙:** DR-01

| 구분 | 조건 |
|---|---|
| **성공** | Given: 사용자가 미사용 이메일과 8자 이상의 비밀번호를 입력한다<br>When: 가입 요청을 제출한다<br>Then: 계정이 생성되고 201 응답이 반환된다 |
| **실패 — 이메일 중복** | Given: 이미 가입된 이메일이 존재한다<br>When: 동일 이메일로 가입 요청한다<br>Then: 409 Conflict 오류와 "이미 사용 중인 이메일입니다" 메시지가 반환된다 |
| **실패 — 비밀번호 미충족** | Given: 7자 이하 비밀번호를 입력한다<br>When: 가입 요청을 제출한다<br>Then: 400 Bad Request 오류와 "비밀번호는 8자 이상이어야 합니다" 메시지가 반환된다 |

### UC-02 로그인

**연관 규칙:** DR-01

| 구분 | 조건 |
|---|---|
| **성공** | Given: 가입된 이메일과 일치하는 비밀번호를 입력한다<br>When: 로그인 요청을 제출한다<br>Then: Access Token과 Refresh Token이 발급된다 |
| **실패 — 이메일 없음** | Given: 가입되지 않은 이메일을 입력한다<br>When: 로그인 요청을 제출한다<br>Then: 401 Unauthorized 오류가 반환된다 |
| **실패 — 비밀번호 불일치** | Given: 가입된 이메일과 틀린 비밀번호를 입력한다<br>When: 로그인 요청을 제출한다<br>Then: 401 Unauthorized 오류가 반환된다 |

### UC-03 할일 관리 (생성·수정·삭제)

**연관 규칙:** DR-01, DR-02, DR-07, DR-08

| 기능 | 구분 | 조건 |
|---|---|---|
| **생성** | 성공 | Given: 인증된 사용자가 제목(필수)과 선택 정보를 입력한다<br>When: 할일 생성 요청을 제출한다<br>Then: 할일이 저장되고 201 응답이 반환된다 |
| | 실패 — 제목 없음 | When: 제목 없이 요청한다<br>Then: 400 Bad Request가 반환된다 |
| | 실패 — 제목 초과 | When: 101자 이상의 제목으로 요청한다<br>Then: 400 Bad Request가 반환된다 |
| **수정** | 성공 — 완료 처리 | When: 완료 여부를 true로 변경한다<br>Then: 할일 상태가 즉시 "완료"로 전환된다 |
| | 성공 — 완료 취소 | When: 완료 여부를 false로 변경한다<br>Then: 마감일이 남은 경우 "진행 중", 지난 경우 "기간 초과"로 전환된다 (DR-08) |
| | 실패 — 타인 데이터 | When: 다른 사용자의 할일 ID로 수정 요청한다<br>Then: 403 Forbidden이 반환된다 |
| **삭제** | 성공 | When: 본인 할일을 삭제 요청한다<br>Then: 할일이 즉시 목록에서 제거된다 (Hard Delete) |
| | 실패 — 타인 데이터 | When: 다른 사용자의 할일 ID로 삭제 요청한다<br>Then: 403 Forbidden이 반환된다 |

### UC-04 할일 목록 조회 및 필터링

**연관 규칙:** DR-01, DR-02, DR-03, DR-04, DR-07

| 구분 | 조건 |
|---|---|
| **상태 필터 — 기간 초과** | Given: 마감일이 현재 시각 이전이고 완료 여부가 false인 할일이 존재한다<br>When: 상태 = "기간 초과" 필터를 적용한다<br>Then: 해당 조건의 할일만 반환된다 |
| **상태 필터 — 진행 중(null 마감일)** | Given: 마감일이 null이고 완료 여부가 false인 할일이 존재한다<br>When: 상태 = "진행 중" 필터를 적용한다<br>Then: 해당 할일이 결과에 포함된다 |
| **카테고리 필터** | When: 특정 카테고리 ID로 필터를 적용한다<br>Then: 해당 카테고리 소속 할일만 반환된다 |
| **타인 데이터 격리** | When: 인증된 사용자가 목록을 조회한다<br>Then: 타 사용자의 할일은 결과에 포함되지 않는다 |

### UC-05 카테고리 관리 (생성·수정·삭제)

**연관 규칙:** DR-01, DR-02, DR-05, DR-06

| 기능 | 구분 | 조건 |
|---|---|---|
| **생성** | 성공 | When: 사용자가 고유한 이름으로 생성 요청한다<br>Then: 카테고리가 생성되고 201 응답이 반환된다 |
| | 실패 — 이름 중복 | Given: 동일 사용자에게 같은 이름의 카테고리가 존재한다<br>When: 동일 이름으로 생성 요청한다<br>Then: 409 Conflict가 반환된다 |
| **삭제** | 성공 — 할일 보존 | When: 카테고리를 삭제한다<br>Then: 소속 할일의 카테고리 ID가 null로 변경되고, 할일 자체는 삭제되지 않는다 (DR-06) |

## 6. 의사결정 로그 (Architecture Decision Records)

| # | 결정 사항 | 이유 | 결정일 |
|---|---|---|---|
| ADR-01 | 카테고리 삭제 시 할일을 함께 삭제하지 않고 카테고리 ID를 null로 처리한다 | 할일은 카테고리와 독립적인 데이터이며, 카테고리 실수 삭제 시 할일 데이터 손실을 방지하기 위함 | 2026-04-28 |
| ADR-02 | 할일 상태(진행 중/완료/기간 초과)를 DB 컬럼이 아닌 조회 시 계산값으로 처리한다 | 마감일 변경 시 상태 컬럼 동기화 오류를 방지하고, 단일 진실 공급원(완료 여부 + 마감일)을 유지하기 위함 | 2026-04-28 |
| ADR-03 | 할일 삭제는 Hard Delete(물리 삭제)로 처리한다 | 개인 할일 관리 앱의 특성상 감사 로그가 불필요하며, 데이터 최소화 원칙을 따름. 향후 요구사항 발생 시 Soft Delete로 전환 가능 | 2026-04-28 |
| ADR-04 | 인증 방식으로 JWT를 채택한다 | 서버 세션 저장소가 불필요한 Stateless 구조를 통해 수평 확장을 용이하게 하기 위함 | 2026-04-28 |

## 7. 확장성 및 경계

### 7.1 확장성 고려
- **메타데이터:** 향후 태그(Tags)나 우선순위(Priority) 필드를 추가할 수 있도록 엔티티 구조를 유연하게 유지한다.
- **다국어:** 카테고리 이름 및 할일 제목은 유니코드를 지원한다.

### 7.2 제외 범위 (현재 버전)
- 팀·공유 기능 (개인 전용 서비스)
- 외부 서비스 연동 (구글 캘린더 등)
- 실시간 푸시 알림
